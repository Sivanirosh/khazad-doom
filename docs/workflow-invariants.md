# Workflow invariants

These invariants define the daemon-owned workflow behavior that v0.1.0 release-polish refactors must preserve. They are intentionally phrased as testable contracts; changing one is a public workflow behavior change, not a cleanup.

## Product doctrine

- **D1 — Pi-first commitment.** Pi is the only real worker harness. `FakeRunner` stays as the deterministic test double, justified by testing, not portability. Daemon state remains harness-neutral JSON; worker execution is Pi-native.
- **D2 — Truthful environmental failure.** Deterministic environment/launch failures block immediately with operator guidance. They must not burn retries or masquerade as implementation failures.
- **D3 — Escalation over termination.** A worker that hits a `must_ask_if` condition should call the shipped worker-only `ask_operator` Pi tool, pause in `awaiting_operator`, and supply its original recommendation/rationale plus bounded-authority and reversible attestations. The daemon owns the 60-second deadline and atomically chooses the first committed operator answer or an eligible exact-option timeout recommendation. Missing, malformed, non-option, ineligible, unavailable, hard-authorization, or unsafe recommendations retain the existing `ask-user` blocked output.
- **D4 — Versioned coupling only.** Khazad-Doom couples to Pi's documented, versioned surfaces such as CLI flags, JSON event streams, and exit codes; it must not depend on Pi internals. `src/pi_contract.rs` is the only module that may parse Pi stdout/stderr or recognize Pi event/error strings; the current contract inventory is `docs/design/pi-contract-inventory.md`. A Pi behavior change may degrade observability, but daemon-owned state remains authoritative for correctness. Unknown fields/events from Pi are tolerated and surfaced as bounded warnings.
- **D5 — Single verification owner.** The daemon owns verification, gates, economics, and attestation. Workers produce evidence claims; daemon checks/gates or human review attest them.
- **D6 — Feedback stays daemon-owned and explicit.** Operators must be able to discover progress and needs-attention states through `status`, `watch`, `monitor`, and `attend`; those surfaces show the same daemon-side feed projection, attention items, deadlines, and exact commands. Herdr may be the optional-default live cockpit for visible workspaces/panes when available, but it is a painter/launcher over daemon-owned state, not a workflow owner. Any Pi feedback adapter is a thin bridge over daemon feed and Herdr focus/open actions; it must not emulate a full live cockpit or replace core monitoring.
- **D7 — Versioned cockpit coupling only.** Khazad-Doom may couple to Herdr only through documented Herdr CLI/session/workspace/tab/pane/agent surfaces wrapped by the `Cockpit` seam. Herdr pane layout, scrollback, and agent-status metadata are observability signals, not daemon truth; Herdr failures may degrade cockpit visibility but must not degrade workflow correctness. Native Herdr-hosted Pi TUI workers are the default when cockpit placement is available; correctness comes only from daemon-owned `submit_worker_result` / `kd_tui_result_artifact` results, or from legacy wrapper stdout/stderr/exit/result artifacts when the wrapper path is explicitly selected or used as fallback. Painters are display-only. The Dashboard pane paints daemon monitor/feed data; bounded operator answers come through the worker pane's Pi `ask_operator` dialog or explicit daemon IPC clients such as `khazad-doom answer`, not an alternate truth store. Pane renames on terminal completion are hygiene only.

Standing rejections:

- Multi-harness worker support is removed from the vision. Revisit only when a concrete second real harness has a user.
- Pi acceptance gates (`attested`, `checked`, `verified`, `reviewed`) are rejected unless daemon verification is retired.
- `fallbackModels` or other silent worker-model failover is rejected unless provider-outage incidents recur and attestation records the actual model per attempt.
- Auto-login and credential mutation are rejected as outside Khazad-Doom's trust boundary; remediation is explicit operator action.

## Roadmap, plan, and finding truth

- Slice JSON plus daemon/run state are the live source of truth for workflow status. Roadmap documents, matrices, and workpackages may summarize or reference that state, but they must not become a competing acceptance or closure ledger. Any roadmap status that cannot be derived from slice/run evidence must be labeled as a planning/audit note rather than accepted workflow truth. `scripts/roadmap-truth-check` enforces that roadmap rows cannot claim done/closed/accepted status without closed slice JSON and named daemon report evidence.
- Implementation work that lands outside Khazad-Doom during normal operation must receive an explicit disposition: Khazad-run, legitimate exemption, freeze or operational exception, or bypass/failure evidence. A silent hand-made product commit is itself workflow-governance evidence until disposed.
- Manual integration disposition: commits from the `kd-20260707-112420-7f061d72` turbulence were manually integrated because that run was superseded by attention/read-model authority defects and was intentionally not resumed. This is a bypass/failure evidence disposition, not evidence that the superseded run completed or was resumed.
- Plan and queue revisions are durable workflow facts, not silent edits. Any accepted change to remaining slice set, order, dependencies, areas, acceptance, verification, or close state records what changed, why, who/what proposed it, who authorized it, and which evidence caused it. Accepted proposal records may grant bounded `authorized_paths`/`action_class` authority to the source slice; worker prompts and daemon path guards honor those grants, not free-form prose. Silent `.workflow/slices/` edits during an active run are not a valid replan mechanism.
- Replan v1 is proposal-only: pending proposals and accepted/rejected/deferred/superseded decisions are daemon state, status/feed render exact decision commands, accepted decisions record whether anything was applied, and the auto-approvable mutation tier is empty.
- Findings that ask for operator intent or propose changes to scope, verification, queue, or workflow policy must reach exactly one terminal disposition: answered, folded into a slice or plan revision, explicitly deferred with revisit condition, or rejected with rationale. Handoffs and reports must not leave such findings as ambient unowned text.

## Run lifetime and ownership

- A run is a durable daemon-owned session keyed by `run_id`. CLI commands and Pi worker tools start, control, or observe that session; they do not define its lifetime.
- No hidden global workflow timeout exists by default. A run continues until it reaches a terminal state, is cancelled, or is marked interrupted/recovered by daemon startup logic.
- Time limits are explicit policy knobs for specific work: verification/gate command timeouts and, when configured, per-worker-attempt timeouts.
- The daemon owns worker prompts, state, worktrees, scheduling, repair, integration gates, cleanup, progress snapshots, status/monitor output, handoff JSON, and artifact inspection.
- Authoritative JSON artifacts are crash-durable only on Unix runtimes. Every replacement uses a same-directory create-new temporary file, complete write plus file synchronization, atomic rename, and parent-directory synchronization; a failure before rename preserves the old complete JSON, while a failure after rename reports uncertain durability with the new complete JSON still readable. Non-Unix runtimes fail closed before an authoritative JSON write rather than silently providing a weaker guarantee. The private legacy-wrapper `status.json` and `exit.json` coordination records invoke this same seam through the daemon binary rather than using shell redirection, so supervision reads only complete snapshots.
- Live `economics.json` is a non-authoritative telemetry projection: SQLite lifecycle state and final reports remain authoritative, and a telemetry write failure cannot change a run outcome. Its successful replacements nevertheless use the same JSON seam, so status/read-model consumers never observe truncated telemetry JSON.

## Slice lifecycle and integration

- JSON Issue Slices are the authoritative work contract. Open slices represent runnable work; closed slices represent accepted historical work.
- JSON Issue Slices are bounded intent contracts, not frozen implementation plans. Acceptance criteria are minimum evidence, not an exhaustive case inventory. Learning is allowed inside the JSON fence; moving the fence requires approval. During an open slice, TDD/code-inspection discoveries directly implied by the slice goal or acceptance and inside declared `areas` may be implemented with the smallest clear change and reported in worker output. Discoveries that require new product intent, public API semantics, dependencies, verification policy, or paths outside `areas` must become `ask-user` blockers or follow-up slices.
- Slice `areas` are repo-relative literal path prefixes, not globs. Directory prefixes should use a trailing slash (for example `src/normia/`); exact files are written as file paths (for example `README.md`). Validation rejects glob characters, parent traversal, absolute paths, leading/trailing whitespace, and leading `./` before workers launch.
- Requested open slices include open dependencies before dependents. Closed dependencies are treated as satisfied and must not launch historical workers again. Explicitly requesting a closed slice is rejected; create a follow-up slice for new work.
- Cycles are invalid across the slice graph.
- Each worker attempt runs one open slice in a daemon-managed isolated worktree. Parallel workers must not share a checkout.
- A completed worker must return valid JSON, commit intended changes, and leave its worktree clean before the daemon may integrate the slice.
- Worker `acceptance_status` entries are evidence claims, not approvals. A worker must not approve its own evidence; daemon-owned checks/gates or later human review attest or reject it separately.
- Independent slices may execute concurrently, but integration into the run branch is serial.
- Run start and resume admission are immediate StateStore transactions. After canonical repository registration and read-only validation, admission persists the run projection and a prepared launch intent before run-owned filesystem, Git reconciliation, cockpit, supervisor-thread, or worker side effects; SQLite triggers independently enforce at most one `pending`/`running` run for the canonical repository id. Setup failures either restore the prior terminal resume state or fail the new run while retaining primary/compensation evidence, so no active-repository claim leaks.
- Launch intents have separate execution epochs and `prepared`, `activated`, and terminal states. A fresh launch records a positive integration-resource ownership claim only after `git worktree add -b` succeeds. Restart never repeats launch side effects: it terminalizes incomplete launch intents and cleans a recorded worktree or deletes an unchanged integration branch only when that durable ownership claim exists. A failed creation or interruption before the claim retains any same-name path/ref as `recovery_required` rather than guessing ownership; moved branches and cleanup/deletion failures are likewise retained. The same reconciliation is idempotent and does not synthesize a worker launch.
- A parallel dependency layer is integration-atomic: Khazad-Doom joins every spawned worker in the active batch/layer and records deterministic per-slice outcomes before deciding whether the layer may merge. No successful worker from a layer is merged after a sibling fails or blocks.
- Every serial slice or integration-repair merge first commits a durable operation journal containing run, kind, slice/repair scope, attempt, exact launch identity when applicable, source branch/commit/tree, expected integration head, the predicted result tree (or the structured prediction failure when Git cannot produce one), and deterministic operation id. Git mutation is followed by one SQLite transaction that marks the journal applied, updates slice or repair-ledger projections, and appends all authoritative completion events.
- Restart reconciles unfinished merge journals before cleanup. Expected head, merge-in-progress state, commit ancestry/parents, source ancestry, result tree, and operation trailer distinguish safe not-started replay, already applied work, conflict/abort evidence, and divergence. Applied work is never merged twice; conflicts and abort failures retain structured files/causes; divergent operator commits are never reset or cleaned and keep the run admitted until explicit reconciliation.
- After each successful integration merge, Khazad-Doom records a checkpoint before advancing, so `resume` can continue from recorded state instead of replaying completed merges.
- Merge conflicts, `ask-user` findings, invalid worker output, dirty worktrees, scope violations, and verification failures become structured blocked/failed artifacts rather than silent best-effort integration.
- The integration gate runs before integration repair. With `integration_repair: "auto"`, repair is only launched after failed gate evidence that is not an operator environment failure; with `"never"`, failed gate evidence is surfaced without repair; with `"always"`, repair may run even after a passing gate for explicit policy reasons. Operator environment failures block with evidence instead of launching repair.
- Repair never bypasses or weakens the gate: whenever repair runs, the daemon reruns the integration gate and only treats the run as successful after the post-repair gate passes.
- Repair workers are not privileged policy mutators. Integration repair may work only inside the already-authorized slice set and gate evidence: changing workflow policy, worker profiles, verification commands, slice contracts, dependencies, or paths outside the authorized areas requires an operator-approved plan revision or follow-up slice.
- Only after a run passes the integration gate may the daemon publish completion. It closes completed slice JSON in the integration branch with `status: "closed"`, `closed_by_run`, and `closed_at`, writes daemon-owned implementation/final report artifacts, and commits exactly an explicit publication manifest containing regular-file close records and reports. Before Git publication, the daemon pins the worktree/repository identity plus every manifest path's exact expected bytes and regular-file mode, requires the transaction's opened root to match that identity, and validates the bytes' run semantics: every slice must decode as its expected ID with `status: "closed"`, matching `closed_by_run`, and nonempty `closed_at`; both reports must contain the same pre-publication summary for that run, record a passed gate, and include every manifested slice ID. On Unix, publication pins the worktree root descriptor and verifies its device/inode against the live pathname throughout the transaction; descriptor-relative `openat` traversal then pins each parent directory and rejects symlinks while reading, so whole-root or parent substitution cannot redirect capture outside the daemon-created worktree. Every publication/recovery transaction also opens and pins the worktree-specific Git administrative directory, common Git directory, and object directory independently, includes those identities in the semantic root manifest, and supplies descriptor-backed `GIT_DIR`, `GIT_COMMON_DIR`, `GIT_WORK_TREE`, object, and index paths to every Git child. The daemon revalidates all four live path attachments before each irreversible phase. It opens the real index and its lock without following either leaf and performs journal replacement relative to the pinned worktree-administration descriptor. Publication writes new objects into a private quarantine, then installs loose objects without following fanout or object leaves relative to the pinned object descriptor; an existing fanout is pinned for the transaction and a detached replacement is rejected. The exact integration-ref parent and any packed-ref leaf are likewise pinned, read without following leaves, and revalidated around a Git-compatible old-parent compare-and-swap run through a private common-directory overlay. That Git-compatible transaction resolves and pins Git's effective configured `core.hooksPath` (including worktree configuration) instead of substituting the default common-directory hooks, and preserves reference hooks and effective reflog policy: enabled branch reflogs get descriptor-confined missing parent directories, disabled reflogs remain disabled, and any unsafe or inaccessible reflog ancestor blocks instead of being downgraded. Reflog-parent creation records operation ownership; a failed ref transaction removes only still-attached, empty directories created by that operation and preserves every pre-existing, replaced, or nonempty directory. `update-ref` and every hook descendant run beneath the authenticated process supervisor, so detached and double-forked hook descendants are reaped before the transaction returns. The daemon snapshots tracked, untracked, ignored, and directory filesystem state, the raw index, and repository/worktree-local configuration immediately around the ref transaction, rejects and exactly restores hook mutations, and gives restoration failure precedence over the original publication failure. On Linux, ordinary ignored build trees remain practical without reading every cached byte: the snapshot records descriptor-confined identity/type/topology/size/mode/time metadata, holds one read lease and descriptor per in-worktree regular-file hard-link group, lazily copies the original bytes through a pinned `0700` backup directory before allowing a hook's write-open to proceed, falls back to an anonymous descriptor-backed memory file if that disk copy cannot be created or completed, and uses the held descriptor when a hook unlinks or replaces a path. Restoration rebuilds changed hard-link groups and metadata exactly; an external hard link, unavailable lease, failure of both lazy-backup targets, or lease-monitor failure blocks publication. If both backup targets fail, the daemon cancels the active authenticated ref transaction, keeps the lease and source descriptors through complete descendant reaping and a bounded fresh-token rollback, and only then releases the lease for restoration; an incomplete rollback retains the durable recovery journal. Rollback restores the original ref representation: an originally loose ref is compare-and-swapped back, while an originally packed-only ref has the transaction-created loose leaf deleted so the unchanged packed ref is exposed without a loose shadow. Thus worktree, linked-worktree metadata, common-directory, nested index, object-fanout, loose-ref-parent, or packed-ref substitution cannot redirect writes into a replacement or detached repository. Publication refuses a pre-staged index, holds the worktree index lock, hashes the semantically pinned bytes with filters disabled into quarantined Git blobs, seeds a private index from the locked real index so unrelated semantic flags survive, constructs the tree against a captured parent, verifies manifest bytes/index/`HEAD` attachment, and advances only the recorded integration ref with an old-parent compare-and-swap. It keeps the index lock through prepared-index installation and final validation, revalidates manifest, index, attachment, and ref after installation, and safely rolls back the ref and original index on a detected final-window input change. A durable index transaction journal contains the exact original and prepared index bytes and fsyncs both the journal and its parent directory before the ref moves; process-loss recovery installs only that journaled prepared index after proving its original index matches the publication parent, and inverse recovery restores only a journaled original whose tree matches a concurrently rewound ref. Stale-journal recovery first takes a nonblocking exclusive claim on the journal and verifies the claimed inode against the lock pathname; divergent refs, unknown index bytes, concurrent recoverers, operator staging without a journal, stale live-owner journals, and failed rollback are blocked with the journal retained. Concurrent index writers cannot enter the transaction; concurrent manifest, attachment, or ref changes cannot redirect or enter the pinned commit. Unrelated tracked, untracked, ignored, filter-created, or semantically flagged files remain untouched and cannot enter the commit. Only after final transaction validation may the daemon advertise the receipt SHA; the receipt records run/slice semantics, parent, tree, exact changed path bytes, and every manifest path/mode/object ID, and the manager validates that receipt against the integration ref, index, semantically pinned manifest, and commit-resident blob semantics before recording publication. Handoff revalidates the durable receipt's parent/tree/path/mode/object identities against the immutable commit and requires the integration ref to remain exactly at that commit; its JSON and transient reports derive `final_sha` from the receipt, and push actions use an explicit receipt-SHA refspec rather than a mutable local branch tip. PR creation requires that same handoff to push successfully and prove the remote branch equals the receipt SHA both before and after creating the PR. On resume, the daemon identifies the exact publication commit from its message and commit-resident manifest object identities, requires the integration ref to equal that exact commit, repairs only a proven journaled crash shape, reconstructs a missing post-commit receipt event when all identities still match, and blocks for operator reconciliation if metadata diverged or the ref advanced beyond it; it must not infer publication identity merely because manifest paths are clean.

## Worker attempt supervision

- Worker launch history is append-only daemon evidence. A persisted `WorkerAttemptLedger` assigns every worker-process launch a unique monotonic `launch_id` and per-slice launch ordinal; retry-budget ordinal, repair ordinal, envelope retry ordinal, and execution/resume epoch are separate fields and never substitute for launch identity.
- Allocation records immutable branch, worktree, and output-stem locations plus `allocated` state and an event in one transaction before any worktree, handoff, artifact, or process-launch side effect. An allocated-but-unlaunched row is retained and reconciled as interrupted after restart; it is never silently reused or deleted.
- Every launch-scoped output, check, handoff, wrapper/TUI spill artifact, activity/status record, and question identity is derived from the immutable launch identity. A retry receives fresh paths and branch/worktree identities; subsequent retries base iterative work on the preceding retained attempt branch rather than overwriting it. Cockpit-to-direct fallback under the same ledger row is allowed only with durable evidence that Pi did not launch; a missing, delayed, or unreadable launch record is ambiguous and must terminalize that row so any later process receives a fresh launch identity.
- Resume advances a durable execution epoch, schedules only unfinished work, and preserves prior ledger rows, raw output, per-launch activity snapshots, branch references, and failure causes. Read models and completion reports expose ledger-first stable history and synthesize clearly marked legacy ordinal-only history per slice with the persisted run start time, without rewriting legacy rows or artifacts.
- Deterministic operator-class worker launch failures, such as Pi provider authentication failures detected by a narrow no-assistant-output plus known stderr signature, block after the first attempt and must not consume the remaining worker retries. Unknown or ambiguous launch failures preserve the existing retry behavior.
- Operator-class launch incidents include `failure_kind`, `retryable`, `operator_action_required`, agent provider/model/profile metadata, and `fix_commands` so status, monitor, reports, and handoffs can surface the same remediation without scraping daemon stderr.
- The effective worker profile is resolved once by the profile module from CLI/env, `.workflow/khazad.json` agent choice, operator-global `~/.khazad-doom/agents.toml`, and built-in defaults. Repo-local `.workflow/agents.toml` is not a runtime input. Pi launch args, `RunnerMetadata`, `profile_summary`, `launch_summary`, source attribution, and auth fix guidance derive from that result; worker surfaces must not assemble provider/model text independently.
- Worker execution is at-least-once, not exactly-once. A timed-out, cancelled, or retried attempt may have produced files or commits in its isolated worktree.
- Worker operator questions are durable daemon state. Ledger-backed `workerAsk`/`workerAskOpen` requests require the exact active `launch_id` and that launch's `KHAZAD_WORKER_TOKEN`; token validation happens in daemon IPC, not only in the Pi extension. An omitted `launch_id` is authorized only by a previously issued legacy run-scoped token, which new ledger-backed launches do not issue. Durable rows preserve launch identity, recommendation, rationale, bounded/reversible inputs, absolute deadline, final answer, and typed answer source. Existing legacy rows/callers without additive fields remain readable and time out/block as before.
- In native TUI mode, `ask_operator` opens the daemon question and prompts in the same worker Pi session, then records the answer through `answerQuestion`. The same-pane cancellation/expiry and headless wait paths return the durable answer/source, including `llm_recommendation_timeout`; they never turn an already auto-answered question back into `timed_out`. The Pi monitor bridge is read-only for questions; CLI/dashboard answer commands remain explicit fallback/debug paths.
- Timeout fallback is daemon-only and eligible only for one exact non-empty declared option with non-empty rationale and true bounded-within-current-slice-or-mission-authority plus reversible attestations. It must never authorize scope expansion, destructive/irreversible work, credentials/secrets, permissions, release/push/handoff, or work outside the slice/mission envelope. Terminal, interrupted, stale-attempt, token-rejected, or unavailable-channel paths never fabricate an answer.
- Operator-answer, timeout, and eligible-deadline races use one StateStore-owned immediate SQLite transaction that writes the question outcome, authoritative typed event, and progress transition together. Replan dispositions use the same command boundary; proposal state, frontier classification, budget consumption, `replan_proposal_decided`, and any frontier auto-accept audit event commit together. A replan decision timeout is durably deferred with canonical `daemon` / `decision_timeout` attribution and a revisit condition, avoiding a wire-incompatible proposal-state variant. The first commit wins. Exact duplicates return `already_applied_idempotently` without another event; conflicting, stale-launch, and missing commands return typed non-success outcomes without applying the requested decision.
- Every StateStore connection explicitly enables foreign keys and WAL and sets a five-second busy timeout before use. Daemon handlers adapt typed command results; they do not authorize a decision with a separate check-then-write read.
- While a worker is paused in `awaiting_operator`, the slice remains `Running`; no `SliceStatus` enum value is added for questions. Fatal worker-attempt timeout accounting excludes the paused interval so a slow operator answer does not consume a retry.
- Parallel worker cancellation is graceful-first. If a run cancellation or sibling layer failure happens while a parallel batch is active, Khazad-Doom propagates cancellation to active workers and still joins every worker thread before the layer returns. On Unix, direct Pi workers run in a dedicated process group: cancellation sends TERM to the group, waits the configured grace period, then sends KILL and reaps it, so descendants are supervised rather than only the immediate Pi child. Non-Unix keeps its documented child-kill fallback.
- Process liveness and output activity are distinct. `Supervisor: alive` means the daemon still observes the child process, not that semantic progress is guaranteed.
- Quiet-worker warnings are advisory. Missing output alone is not terminal unless an explicit timeout/policy makes it terminal.
- `worker_attempt_timeout_seconds: 0` means no fatal worker-attempt timeout. Any nonzero worker-attempt timeout is an explicit repo/operator policy and applies to an attempt, not to the whole run lifetime.
- `worker_question_timeout_seconds: 0` means pending operator questions wait indefinitely. The built-in and this repository use 60 seconds; any nonzero value becomes a durable daemon-owned absolute deadline shown with the eligible fallback in status/feed/monitor/attend and the same-pane dialog.

## Cancellation, interruption, and resume

- `cancel` requests cancellation through daemon state and worker process signalling; it is an explicit operator action, not a side effect of closing a monitor, status follower, or CLI session.
- Terminalization is a daemon-owned recoverable protocol: a first-commit-wins durable intent transaction interrupts every still-pending worker question, terminalizes every allocated/running launch-ledger row with immutable finish evidence, deactivates active slice attempts, and projects terminal progress; the daemon then atomically replaces `run-summary.json`; one SQLite transaction publishes matching `runs.status`/primary error plus its canonical terminal event; only then does it perform notification bookkeeping and worktree cleanup. A mandatory intent, summary, status, or terminal-event failure leaves the public run nonterminal with the exact durable intent for recovery. Unstarted `pending` slice rows are planned work rather than active attempts and remain pending so an explicit resume can distinguish and schedule them; a resumed attempt never reactivates an old question or its deadline, even if the numeric attempt is reused.
- If daemon startup or explicit resume encounters an incomplete terminal transition, it reconciles that exact daemon-owned intent before considering interruption or new work. It never infers an outcome from renderer, pane, or summary output; canonical terminal events, notification bookkeeping, and cleanup claims are idempotent. Cleanup or visibility failures are non-authoritative incidents and never replace the primary terminal reason.
- If the daemon starts and discovers an active run with no durable terminal intent from a previous process, it first records an `interrupted` intent and then runs the same terminalization protocol.
- `resume` is explicit. It uses durable checkpoint/run state for remaining work, increments the execution epoch, reconciles unlaunched allocations, and never claims to resurrect a lost worker process or reuse its launch identity.

## Verification and gates

- Slice `verify` commands are worker-local checks and must stay within the slice's fix authority. Configured `verify_profile` commands are integration-gate checks only, so broad repo checks run after merge where run-level repair authority can cover the selected slice set. Both command types must run with their declared repo-relative context and environment; the daemon pins the worktree root, opens the command directory by descriptor-relative, no-follow traversal, and uses `fchdir` in the child. On Linux, the same open root device/inode anchors Git inspection, tracked and ignored filesystem capture, cache revalidation, execution, and restoration even if the worktree pathname is replaced; symlink, `..`, whole-root, and validation-to-spawn substitution therefore cannot split command execution from daemon evidence. Optional `worktree_setup` commands are daemon-owned bootstrap steps for each worker/integration worktree, run without verification-cache reuse, and must leave the git worktree clean except ignored files. Missing tools, invalid verify cwd, shell spawn failures, and non-executable commands are classified as daemon/operator environment failures rather than worker auto-fix failures.
- Every slice and integration verification command is bracketed by a daemon-owned, raw-byte/NUL-safe Git and filesystem snapshot that distinguishes resolved `HEAD`, symbolic-versus-detached `HEAD` attachment, semantic index entries plus index version/extensions, staged changes, unstaged changes, untracked files, renames, deletions, nonignored empty directories, tracked file bytes or symlink targets, type, mode, and modification time. Snapshot Git inspections use a private copy through `GIT_INDEX_FILE`; the real index is compared byte-for-byte before and after capture, so daemon inspection cannot refresh the restoration baseline. Snapshot Git also disables fsmonitor. If any configured clean/smudge/process/textconv/external-diff command could execute, unstaged observation uses descriptor-confined filesystem metadata against the raw index stat cache instead of asking Git to convert worktree content; only configurations with no executable conversion path may use Git's semantic raw diff. Pre-command capture, post-command capture, cache revalidation, and restoration therefore never execute a configured content filter. Every daemon-owned Git subprocess in those verification phases also forces hooks off, so HEAD restoration cannot launch an unsupervised reference hook after verifier descendant reaping; publication retains its separately supervised effective-hook behavior. The command environment disables optional Git index refreshes; any remaining command-caused raw index change—including stat-cache, format, extension, or flags—is a real verification mutation, and restoration atomically reinstalls the exact original raw index bytes. Verification requires a clean nonignored prestate; a prestate with worktree-hiding `skip-worktree`/`assume-unchanged` state or a hard-linked tracked file that cannot be restored in isolation is operator-owned and rejected before execution. Tracked snapshot reads, ignored/cache walks, untracked cleanup, and raw restoration open every component relative to already-open directory descriptors and never follow a substituted parent symlink; a parent renamed after opening remains the bounded target, while the concurrent pathname change is surfaced as operator-owned. After restoring the original index and tracked ignore files, cleanup repeatedly rediscovers and removes nonignored files and empty-directory additions until verifier-created ignore rules can no longer hide newly exposed residue; ignored outputs under the original rules may remain. Any mutation during command success, failure, timeout, or cancellation must be restored to the exact prestate—including raw bytes hidden by clean filters and metadata-only changes—and recorded as structured before/after/restored evidence; mutation or restoration failure is an operator-intervention failure and cannot enter retry, repair, merge, publication, or `final_sha`. If after-snapshot capture itself fails, restoration proceeds from the durable pre-command snapshot. The daemon recaptures the whole tracked/Git state and raw filesystem digest immediately before cache reuse and immediately before execution; any concurrent pre-command change blocks as an operator-owned condition without command execution, worker retry, repair, or restoration over that concurrent state. If cancellation and mutation coincide, the daemon persists structured command/check or gate evidence before propagating cancellation; restoration failure takes precedence over ordinary cancellation. Ignored outputs may remain ignored but never become publication inputs.
- On Linux, each verification shell runs under a dedicated daemon-spawned supervisor that is a child subreaper. The supervisor retains the command ancestry independently of environment variables, file descriptors, process groups, or sessions; after normal exit, command failure, timeout, or cancellation it repeatedly terminates and reaps the full descendant tree, including detached, double-forked, environment-scrubbing, inherited-descriptor-closing, and fork-on-termination children, before the `ShellCommand` returns and before the post-command snapshot. Multiple consecutive empty `/proc` scans are required before quiescence, transient disappearing/inaccessible process entries are tolerated without converting a successful command into a scanner race failure, and the command-side inherited filter first kills foreign audit architectures and, on x86-64, x32 syscall-number aliases before it refuses attempts to signal the supervisor while allowing ordinary native-ABI child-process signalling. A CLOEXEC supervision pipe and process-group cleanup remain defense in depth. The hidden supervisor reports internal success or failure through a separate CLOEXEC-protected result descriptor that is closed before the verifier shell is spawned; verifier-controlled stderr and exit codes, including exit 125, can therefore produce only ordinary command evidence and can never forge `process_supervision_failed`. The daemon consumes and validates that private result after supervisor termination on ordinary completion, timeout, cancellation, and the post-exit cancellation race; authenticated failure or a missing/malformed result outranks ordinary timeout/cancellation. Supervisor setup, missing or malformed private results, non-transient scan, or quiescence failures are typed operator failures rather than successful command exits.
- Verification-cache identity includes repository/worktree/cwd identity, verified `HEAD` and tree, clean pre-command snapshot, a raw worktree-filesystem content/type/mode/time digest (excluding only Git administrative state), command configuration, inherited and declared environment, timeout, and policy version. Only a successful command whose tracked snapshot is unchanged and whose filesystem digest is unchanged may populate the cache, and both identities are revalidated again before returning a cache hit; an allowed ignored output or metadata change therefore prevents reuse rather than becoming a stale cached pass.
- Gate command plans preserve profile insertion order, merge exact duplicate commands within a gate, and may fail fast after the first gate failure when `gate_fail_fast` is enabled.
- The workflow manager owns lifecycle ordering, retries, repair decisions, checkpointing, and state transitions; the workflow gate/shell seam owns only command resolution/execution details and returns typed check/gate results.
- Verification and gate timeouts are per-command hang protection. They are not global workflow timeouts and must not be reused to cap total run lifetime.
- Gate failures are reported with command evidence and must be repaired or surfaced as blocked/failed before handoff.
- Terminal `blocked` and `failed` states must include structured primary-reason data, not only prose: reason kind, resolution owner, retryability/operator-action flags where applicable, evidence links, and remediation or disposition links. `status` JSON exposes this as `primary_terminal_reason` and the versioned daemon `feed` mirrors it as `feed.terminal_reason` plus exact `feed.operator_commands`; monitor/watch/Pi renderers paint those blocks instead of reclassifying raw text.
- Final reports and handoff JSON expose explicit `exit_states`, `evidence_attestation`, and `plan_revisions` as read-only summaries of existing lifecycle state. They must not introduce hidden gates, extra worker turns, or a second source of truth. Pending replan proposals block handoff readiness until an operator records an approved disposition; accepted/rejected/deferred/superseded proposals remain visible with evidence and rationale.
- Status/watch/monitor snapshots and final reports include runtime economics: agent calls, daemon-owned command executions, cache hit/miss counts, repair policy/attempts, phase durations, duplicate-command telemetry, and SLA violations.
- Completed runs may still have incidents. Resume events, prior run errors, cleanup issues, integration repairs, and non-fatal lifecycle warnings must remain visible as run incidents instead of being hidden by a final `completed` status.
- Every terminal run writes `.workflow/runs/<run>/outputs/run-summary.json` before daemon worktree cleanup and before the terminal state is advertised. Failed/cancelled/blocked summaries retain primary failure or cancel reason plus bounded worktree/attempt diagnostics where available; they are not committed reports.
- If run start records an optional `.workflow/runs/<run>/origin.json` target, terminal feedback is emitted only after that terminal summary exists, only for completed/blocked/failed/cancelled transitions, and only as inert declarative evidence through the Cockpit Herdr send seam. Dedupe records are per terminal transition so a resumed run can separately report blocked and completed while retry paths do not duplicate one transition. Pending worker questions and replan decisions may also send/focus visibility-only attention messages through `herdr agent send` / `herdr agent focus`; attention records are keyed by concrete question/proposal ids and payloads carry only reason text plus exact daemon `answer`/`replan` and `status`/`monitor` commands. Failures are recorded as incidents only. Runs without an origin target do not notify. Missing Herdr, malformed/stale recorded targets, or send/focus failures are warning-level visibility evidence and must not affect status, verification, merge, handoff readiness, or final SHA. Interrupted is excluded in v1 because daemon-restart recovery can stale the origin target.
- Run start is clean-by-default: the source repo dirty state and base branch/SHA are captured in `preflight.json`, and dirty starts require explicit `--allow-dirty`.

## Progress, status, and monitor state

- The daemon/state store is the source of truth for run status, slice states, events, and live progress snapshots.
- `status`, `watch`, and `monitor` render the same daemon state. They must not own workflow state or infer cancellation from UI/session shutdown.
- Status interpretation is centralized daemon-side in the status feed projection. Renderers are painters: they may choose layout/color, but not invent different wording or re-interpret daemon event payloads independently. The CLI monitor/watch/attend paths and explicit Pi `/khazad-attach` / `/khazad-explain` adapter actions paint `RunDetails.feed`; when feed data is unavailable, they report feed unavailability rather than rebuilding workflow wording from raw events. The `feed.attention` list is first-class and must render full, untruncated pending-question/proposal text, options, exact commands, and deadlines.
- `monitor --latest` must not make terminal runs disappear. When no active run exists, it keeps the latest terminal run summary visible, including incidents and handoff readiness.
- Progress output may distinguish supervisor liveness, worker process state, last output event, human last-semantic-progress summary, configured timeouts, and advisory quiet-worker warnings.
- When a parallel worker layer is active, status/watch/monitor output exposes the layer explicitly and lists the active slice IDs in deterministic order.
- The Herdr cockpit adapter, when enabled, may open/focus run workspaces with a Dashboard pane and deterministically named native Pi TUI worker agents tied to run id, slice id, and immutable launch id; retry, repair, and envelope ordinals remain separate display context. The Dashboard paints daemon monitor/feed data; operator attention uses the worker pane's `ask_operator` Pi dialog or daemon commands such as `khazad-doom attend`; successful native worker panes remain available through terminal-summary hygiene so they can be renamed with outcome markers. Herdr absence, cockpit startup failure, native TUI startup failure, activity-painter exit, unexpected pane rename failure, or wrapper handoff failure before Pi launches falls back to direct execution or degrades visibility only by default and records/displays non-fatal evidence; it must not by itself change run/slice status, worker authorization, verification, merge, or handoff readiness. The Planner Pi pane remains explicitly deferred until RPL planner authority exists.
- Cockpit layout pane identities are not durable authority. Every layout operation resolves a live anchor by role at use time: worker-region placeholder, stable worker slot label, unlabeled fresh root, or Dashboard as a last-resort split base. Closed pane ids must not be treated as empty available panes, and a workspace whose worker region was emptied by timeout/cancel/retry cleanup must remain placeable for the next worker attempt. This invariant is backed by COCKPIT-ANCHOR-01 evidence (`docs/design/evidence/cockpit-anchor-stale-pane-2026-07-08.md`).
- Herdr worker panes/agents are not a freeform authority channel. Native TUI workers must submit through the daemon-owned `submit_worker_result` artifact contract; legacy wrapper workers write stdout/stderr/exit/result artifacts under `.workflow/runs/<run>/outputs/` and feed the same daemon worker-attempt supervision. The daemon never reads worker JSON from pane text, scrollback, or Herdr agent-status metadata. Normal operator control means observe, focus, answer bounded `ask_operator` dialogs, and request daemon-owned actions such as cancel or answer; arbitrary terminal typing into the worker pane is not accepted workflow input, and any manual takeover must be explicit evidence, not silent accepted worker output.
- The Pi feedback adapter is a thin bridge over daemon commands/data and Herdr open/focus actions: start/shape via the CLI, explain from `RunDetails.feed`, summarize handoff JSON, answer blockers through daemon `answerQuestion`, and delegate Herdr focus/open to `khazad-doom cockpit open`. It cleans up all session-bound resources on Pi session replacement/reload, never infers run lifecycle from Pi sessions, never renders a full live multi-agent cockpit, and is never required for core monitoring or headless operation.

## Artifacts, handoffs, and remotes

- Runtime handoffs, raw worker outputs, checkpoints, and inspection artifacts live under `.workflow/runs/` and remain transient/gitignored unless explicitly promoted elsewhere.
- Worker handoff JSON is generated before the worker starts and records the exact slice contract, worktree path, branch, run id, and output path the worker must use.
- `inspect` and blocked/failed artifacts expose bounded diagnostics without requiring maintainers to scrape daemon internals.
- `khazad-doom handoff` prints branch, summary, and suggested push/PR commands by default. It must not mutate remotes unless `--push`, `--create-pr`, or explicit repository configuration requests that behavior; `--dry-run` suppresses configured actions.
